# 用户状态标签定义详解

本文档详细说明"用户状态总览"表中各项标签的判定标准、优先级和逻辑。

---

## 一、用户状态（state_4）

### 概述
用户状态有 7 种，分别从以下维度判定：
- **最后投注日期** (last_bet_day)
- **最后充值成功日期** (last_recharge_day)  
- **距今天数间隔** (b_gap, r_gap)

### 定义与优先级

**优先级从高到低（以下顺序判定，匹配即止）：**

| 优先级 | 状态 | 条件 | 说明 |
|-------|-----|------|------|
| 1 | **未激活** | `last_bet_day IS NULL AND last_recharge_day IS NULL` | 从未投注且从未充值 |
| 2 | **只充不玩** | `last_bet_day IS NULL AND last_recharge_day IS NOT NULL` | 曾充过值但从未投注 |
| 3 | **只玩不充** | `last_bet_day IS NOT NULL AND last_recharge_day IS NULL` | 曾投注过但从未充值 |
| 4 | **健康稳定** | `b_gap < stable_b AND r_gap < stable_r` | 近期既投注又充值 |
| 5 | **已流失** | `b_gap >= lost_b AND r_gap >= lost_r` | 长期未投注且未充值 |
| 6 | **高危流失** | `b_gap >= stable_b AND r_gap >= stable_r` | 中期未投注且未充值（风险中等） |
| 7 | **变慢预警** | 其他情况 | 活跃度开始下降的过渡状态 |

### 状态判定阈值

配置表 `ops_config.thresholds` 中的参数：

| 参数 | 默认值 | 含义 |
|-----|-------|------|
| `stable_b_gap_threshold` | 2 天 | "健康稳定"的投注间隔阈值 |
| `stable_r_gap_threshold` | 5 天 | "健康稳定"的充值间隔阈值 |
| `lost_b_gap_threshold` | 5 天 | "已流失"的投注间隔阈值 |
| `lost_r_gap_threshold` | 10 天 | "已流失"的充值间隔阈值 |

### 状态子类（sub_state）

仅在用户既投注过又充值过的情况下，细分非健康状态：

| 状态子类 | 条件 |
|---------|------|
| **还玩但不充** | `b_gap < stable_b AND r_gap >= stable_r` |
| **还充但不玩** | `b_gap >= stable_b AND r_gap < stable_r` |

**示例：** 用户今天投注过，但 10 天没充过值 → "高危流失"+"还玩但不充"

## 二、衰减情况（reason_code）

### 定义

判定用户是否在衰减及衰减类型。

### 判定条件

| 衰减情况 | 条件 | 计算依据 |
|---------|------|--------|
| **投注衰减** | `bet_drop < bet_drop_threshold` 且 `rech_drop ≥ rech_drop_threshold` | 投注金额/天数相对下降 ≥ 60% |
| **充值衰减** | `rech_drop < rech_drop_threshold` 且 `bet_drop ≥ bet_drop_threshold` | 充值金额/天数相对下降 ≥ 60% |
| **双衰减** | `bet_drop < bet_drop_threshold` 且 `rech_drop < rech_drop_threshold` | 投注和充值都下降 ≥ 60% |
| **稳定** | 其他情况 | 无明显衰减 |

### 衰减比计算

**bet_drop（投注衰减比）：**
$$\text{bet\_drop} = \frac{\text{短窗期投注}/\text{短窗天数}}{\text{长窗期投注}/\text{长窗天数}}$$

- 短窗期：最近 7 天（可配置 `bet_drop_short_days`）
- 长窗期：最近 30 天（可配置 `bet_drop_long_days`）
- 如果 bet_drop = 0.5，表示最近一周的投注强度是最近一月的 50%（下降 50%）

**rech_drop（充值衰减比）：** 同理

### 默认阈值

| 参数 | 默认值 |
|-----|-------|
| `bet_drop_threshold` | 0.6（60%） |
| `rech_drop_threshold` | 0.6（60%） |
| `bet_drop_short_days` | 7 天 |
| `bet_drop_long_days` | 30 天 |

## 三、资金行为类型（funding_type）

### 定义

判定用户提现占充值的比例，区分不同的资金流向特征。

### 判定条件

| 资金行为类型 | 条件 | 说明 |
|-----------|------|------|
| **资金回收型** | `wd_rate_long ≥ wd_rate_long_threshold` | 长期提现占充值 ≥ 80%，说明在持续提现 |
| **正常循环** | 其他情况 | 提现相对较少，资金留存在平台 |

### 提现比例计算

**wd_rate_short（短窗口提现比例）：**
$$\text{wd\_rate\_short} = \frac{\text{提现成功金额}}{\text{充值成功金额}}$$
  - 窗口：最近 7 天（可配置）

**wd_rate_long（长窗口提现比例）：**
$$\text{wd\_rate\_long} = \frac{\text{提现成功金额}}{\text{充值成功金额}}$$
  - 窗口：最近 30 天（可配置）

### 默认阈值

| 参数 | 默认值 |
|-----|-------|
| `wd_rate_long_threshold` | 0.8（80%） |
| `wd_short_days` | 7 天 |
| `wd_long_days` | 30 天 |

## 四、盈亏原因标签（profit_reason_tag）

### 定义

基于用户的盈亏状况和流失/衰减特征，判定其离场的原因。

### 判定条件

| 盈亏原因标签 | 条件 |
|-----------|------|
| **亏损型衰减** | `p_short < 0` 且 (`reason_code ≠ '稳定'` 或 `state_4 ∈ ('变慢预警', '高危流失')`) |
| **盈利型退出/回收** | `p_short > 0` 且 `funding_type = '资金回收型'` |
| **高波动型** | `p_long ≠ NULL` 且 `((p_short ≥ 0 AND p_long < 0) OR (p_short < 0 AND p_long ≥ 0))` |
| **均衡型自然衰减** | 其他情况（包括无盈亏数据） |

### 盈亏计算

**p_short（短窗口盈亏）：**
$$\text{p\_short} = \text{派彩金额} - \text{投注金额}$$
  - 窗口：最近 7 天

**p_long（长窗口盈亏）：**  
$$\text{p\_long} = \text{派彩金额} - \text{投注金额}$$
  - 窗口：最近 30 天

### 说明

- **亏损型衰减**：用户在亏损且活跃度下降，有可能是因为经历了连续亏损导致流失
- **盈利型退出/回收**：用户虽然盈利，但在持续提现，是主动离场或风险规避
- **高波动型**：短期和长期盈亏方向相反，表现不稳定
- **均衡型自然衰减**：其他衰减原因，可能是自然周期或其他因素

## 五、支付摩擦标签（pay_friction_tag）

### 定义

判定用户在充值支付环节是否遇到障碍。

### 判定条件

| 支付摩擦标签 | 条件 |
|-----------|------|
| **支付摩擦-急** | `pay_fail_rate_short ≥ pay_friction_fail_rate_threshold` |
| **支付摩擦-持续** | `pay_fail_rate_long ≥ pay_friction_fail_rate_threshold` |
| **支付正常** | 其他情况 |

### 支付失败率计算

**pay_fail_rate_short（短窗口失败率）：**
$$\text{pay\_fail\_rate\_short} = \frac{\text{充值失败次数}}{\text{充值总尝试次数}}$$
  - 窗口：最近 1 天（可配置 `pay_friction_short_days`）

**pay_fail_rate_long（长窗口失败率）：**
$$\text{pay\_fail\_rate\_long} = \frac{\text{充值失败次数}}{\text{充值总尝试次数}}$$
  - 窗口：最近 3 天（可配置 `pay_friction_long_days`）

### 优先级

1. 先判 **短窗口失败率** → "支付摩擦-急"（新出现的支付问题）
2. 再判 **长窗口失败率** → "支付摩擦-持续"（长期支付问题）
3. 都不满足 → "支付正常"

### 默认阈值

| 参数 | 默认值 |
|-----|-------|
| `pay_friction_fail_rate_threshold` | 0.5（50%） |
| `pay_friction_short_days` | 1 天 |
| `pay_friction_long_days` | 3 天 |

## 六、工作流与优先级总结

### 完整判定流程

```
1️⃣ 计算基础特征
   ├─ last_bet_day, last_recharge_day
   ├─ b_gap, r_gap

2️⃣ 计算滑窗指标（短窗 7 天 / 长窗 30 天）
   ├─ bet_drop, rech_drop
   ├─ wd_rate_short, wd_rate_long
   ├─ p_short, p_long
   ├─ pay_fail_rate_short, pay_fail_rate_long
   ├─ bonus_rate_rech, bonus_rate_bet

3️⃣ 判定各标签
   ├─ state_4（用户状态）    ← 优先级最高
   ├─ reason_code（衰减情况）
   ├─ funding_type（资金行为类型）
   ├─ pay_friction_tag（支付摩擦）
   ├─ profit_reason_tag（盈亏原因）← 依赖前面多个标签
```

### 关键依赖关系

- **state_4** 最先判定（独立）
- **reason_code** 依赖 bet_drop, rech_drop
- **funding_type** 依赖 wd_rate_long
- **pay_friction_tag** 依赖 pay_fail_rate
- **profit_reason_tag** 依赖前面全部标签和 p_short, p_long

### 存在多条件匹配时的处理

采用 **按顺序判定、匹配即止** 的策略（通过 CASE WHEN 实现）：

```sql
CASE
  WHEN 条件1 THEN '标签A'
  WHEN 条件2 THEN '标签B'
  WHEN 条件3 THEN '标签C'
  ELSE '默认标签'
END
```

## 七、配置参数修改方式

所有阈值都存储在 `ops_config.thresholds` 表中，格式为 (key, value) 对：

```sql
-- 查看所有配置
SELECT * FROM ops_config.thresholds;

-- 修改投注衰减阈值（从 60% 改为 50%）
UPDATE ops_config.thresholds 
SET value = 0.5 
WHERE key = 'bet_drop_threshold';

-- 修改后需要重新计算标签
python -m autotag.model.labeling --dt 2026-03-05
```

## 八、补充说明

### 空值处理

- 如果用户从未投注 (`last_bet_day = NULL`)，则 `bet_drop = NULL`
- 如果没有提现记录 (`wd_amt = 0`)，则 `wd_rate = NULL`
- 标签中如果依赖的指标为 NULL，该标签也会是 NULL 或使用默认值

### 聚合维度

所有标签都是**按 (user_id, as_of_date) 聚合**的：
- 一个用户在同一天只有一行记录
- 每天重新计算一次标签

### 时间定义

- **as_of_date**：统计日期（数据查询的截止日期）
- **biz_date**：业务日期（事件发生的日期）
- **窗口期计算**：包含 as_of_date 本身

## 九、实际查询示例

```sql
-- 查看今天所有用户的标签
SELECT 
  统计日期,
  用户ID,
  用户状态,
  衰减情况,
  资金行为类型,
  盈亏原因标签,
  支付摩擦标签
FROM ops."用户状态总览"
WHERE 统计日期 = CURRENT_DATE;

-- 查看"已流失"的用户
SELECT * FROM ops."用户状态总览"
WHERE 用户状态 = '已流失'
  AND 统计日期 = CURRENT_DATE;

-- 查看有支付摩擦的用户
SELECT * FROM ops."用户状态总览"
WHERE 支付摩擦标签 LIKE '支付摩擦%'
  AND 统计日期 = CURRENT_DATE;
```
